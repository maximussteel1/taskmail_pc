# Task Manager 分阶段实施路线（v0.1）

> 文档层级：Layer 3（未来平台实施路线）
>
> 重要说明：该路线图针对未来 Task Manager 平台建设顺序，不等同于当前邮件适配层仓库的阶段状态。

## 1. 文档目的

本文件用于定义 Task Manager 平台的实施顺序与阶段边界。

目标不是一次性搭建完整平台，而是在保证长期可扩展性的前提下，以最小闭环方式逐步建设。平台开发应遵循“先稳定底座，再扩展能力；先打通单条任务链路，再扩展工具面”的原则。

---

## 2. 实施总原则

### 2.1 接口先行，后端从简

优先冻结工具协议、任务协议和返回结构，再实现最小后端。

首期实现可以很简单，但接口必须为后续扩展预留空间。

### 2.2 先做一条真实闭环，不做泛化系统

首期只围绕一条高价值任务链路建设，例如：

“从历史文档中检索某主题资料，读取关键片段，生成带引用摘要，并记录任务状态。”

只有这条链路稳定后，才继续增加新工具族。

### 2.3 先做只读记忆，不做自动写回知识库

外部记忆系统首期只负责：

- 文档入库
- 文档检索
- 文档片段读取
- 引用定位

不在首期开放“自动总结后写回知识库”之类能力，以避免错误积累。

### 2.4 CLI 先统一，MCP 后接入

首期以统一 CLI 作为执行层，先稳定本地接口契约。

当工具协议成熟后，再封装为 MCP / 函数工具暴露给 Codex。不要在首期同时调试 CLI、MCP 和复杂调度逻辑。

### 2.5 先固定命名空间，再扩动作集合

优先固定域和命名，例如：

- `memory.*`
- `task.*`
- `doc.*`
- `sheet.*`
- `data.*`
- `email.*`

首期只实现少量动作，不追求动作覆盖面。

---

## 3. 总体建设顺序

推荐建设顺序如下：

1. 冻结基线文档与 schema
2. 跑通 memory 最小能力
3. 跑通 task state 最小能力
4. 实现第一条调度闭环
5. 增加一个高价值工具族
6. 引入评估、日志和回归测试
7. 再扩展到 MCP 与更多工具

这个顺序的核心思想是：

**先证明系统能稳定完成一个任务，再证明它能扩展。**

---

## 4. 分阶段实施路线

## 阶段 0：基线冻结

### 目标

把“平台规则、工具契约、最小 schema”固定下来，作为后续开发基线。

### 本阶段产出

- 《平台设计总纲》
- 《平台落地方案》
- 《工具接口规范》
- 首批 JSON Schema
- 工具命名空间定义
- 仓库目录结构约定

### 本阶段只做什么

- 确定命名
- 确定 JSON 包络
- 确定错误码族
- 确定核心接口

### 本阶段不做什么

- 不做复杂调度器
- 不接入过多工具
- 不做完整知识库清洗
- 不追求前端界面

### 验收标准

- 核心 schema 可校验
- `memory.search / memory.read / task.get / task.patch` 的输入输出结构固定
- CLI 命名空间不再频繁变化

---

## 阶段 1：外部记忆系统 MVP

### 目标

建设一个最小可用的文档记忆后端，支持 Task Manager 调用。

### 建议范围

只选择一小批高价值文档进入首期，例如：

- 某一类历史报告
- 某一项目完整资料
- 某一类经常复用的技术文档

不要一次性导入全部历史文件。

### 首期功能

- 文档登记
- 基础元数据抽取
- 文档分块
- `memory.search`
- `memory.read`
- 引用定位信息返回

### 推荐首期元数据字段

- `doc_id`
- `title`
- `doc_type`
- `project`
- `date`
- `tags`
- `version`
- `source_path`

### 检索实现策略

首期实现可以简单，例如：

- 元数据过滤
- 关键词检索
- 基础相关性排序

但接口层应预留未来扩展位，例如：

- `mode`
- `sort`
- `semantic_enabled`
- `date_range`

也就是说：

**实现可以简单，接口不能太短视。**

### 验收标准

- 能稳定返回候选文档片段
- 返回结果包含 `doc_id / chunk_id / locator`
- 可以读取指定片段原文
- 可以支撑“带引用摘要”任务

---

## 阶段 2：任务状态系统 MVP

### 目标

建立任务状态外部存储，让 Task Manager 不依赖模型上下文保存任务进度。

### 首期功能

- `task.get`
- `task.patch`
- 任务基础状态记录
- 中间结果摘要保存
- 来源记录

### 首期任务状态建议字段

- `task_id`
- `goal`
- `status`
- `input_refs`
- `selected_sources`
- `intermediate_notes`
- `outputs`
- `updated_at`

### 验收标准

- 任一任务都能通过 `task_id` 找回当前状态
- 中间摘要和来源记录能独立于模型保存
- 调度器可在新实例中继续任务

---

## 阶段 3：首条调度闭环

### 目标

让 Task Manager 跑通第一条真实业务链路。

### 推荐首条闭环

用户请求：

“检索项目 A 中关于设备 X 校准条件的历史资料，生成带引用摘要。”

### 标准流程

1. Task Manager 创建任务
2. 调用 `memory.search`
3. 读取 Top-K 片段
4. 去重与筛选来源
5. 生成摘要
6. 附上引用信息
7. 调用 `task.patch` 更新状态

### 本阶段边界

- 不允许自由多轮自发扩展任务
- 不允许无限工具循环
- 不引入自动知识写回
- 不接入第二个业务域

### 验收标准

- 闭环可重复运行
- 输出带来源定位
- 任务中断后可恢复
- 错误可定位到具体步骤

---

## 阶段 4：增加第一个生产工具族

### 目标

在记忆闭环稳定后，只增加一个最有价值的执行域。

### 建议优先级

优先从以下两类中选一个：

#### 方案 A：doc.*

适用于你当前“资料检索 + 汇总 + 初稿生成”需求最强的情况。

首期动作可以包括：

- `doc.generate_outline`
- `doc.generate_draft`
- `doc.merge_sections`

#### 方案 B：sheet.*

适用于你当前“表格清洗、字段映射、标准格式导出”需求最强的情况。

首期动作可以包括：

- `sheet.validate_headers`
- `sheet.transform`
- `sheet.export_standard`

### 原则

首期只选一个域，不同时接入 doc 与 sheet 两大族。

### 验收标准

- 新工具与既有 JSON 协议兼容
- 调度层无须重写，只需新增工具映射
- 至少跑通一条“memory + 新工具”联合任务

---

## 阶段 5：评估、日志与回归

### 目标

在工具数量变多之前，把质量控制能力补上。

### 必补内容

- 工具调用日志
- 错误码与错误上下文
- 示例数据集
- 回归测试样例
- schema 校验测试
- 最小评估集

### 为什么这一阶段必须尽早补

因为一旦工具数开始增加，没有日志和回归测试，系统会越来越难维护。

### 验收标准

- 每个工具调用都可追踪
- 常见失败可定位
- schema 变更可检测影响面
- 关键闭环可回归复测

---

## 阶段 6：MCP / Codex 工具暴露

### 目标

在 CLI 和协议稳定后，再把工具以更正式的方式暴露给 Codex。

### 推荐顺序

1. 先稳定本地 `tm` CLI
2. 再做工具目录与工具描述
3. 最后封装为 MCP 或函数工具

### 原则

Codex 使用的是类型化工具能力，而不是直接依赖随意 shell 命令。

CLI 是执行层，MCP / function tool 是代理层。

### 验收标准

- Codex 可发现核心工具
- 工具描述和 schema 一致
- 调度层不依赖人工补提示词记忆工具细节

---

## 5. 为扩展性预留但首期不实现的内容

以下内容建议在 schema 和命名上预留，但不要首期就全部实现：

- 语义检索 / 混合检索
- 文档版本比较
- 自动标签建议
- 多项目隔离
- 权限分级
- 工具注册中心
- 审计视图
- 多任务并行编排
- 自动知识写回
- 邮件与日历联动

原则是：

**接口可以预留，能力不要抢跑。**

---

## 6. 当前最不该做的事

为避免系统失控，当前阶段不建议做以下事情：

### 6.1 不要一次导入全部历史文档

首期只选一个文档集合试点。

### 6.2 不要把大量业务知识塞进提示词

提示词只负责规则，不负责长期记忆。

### 6.3 不要先做通用智能代理循环

首期调度应尽量固定流程，而不是放任模型无限规划。

### 6.4 不要同时做多个工具域

同一阶段只推进一个新工具族。

### 6.5 不要先做漂亮前端

平台早期价值来自闭环能力，而不是界面。

---

## 7. 当前建议的实际推进顺序

结合当前已有文档与 schema，建议你接下来按以下顺序推进：

### 第一步

冻结现有三份设计文档与首批 schema，作为 v0.1 基线。

### 第二步

实现最小 `tm` CLI 骨架，只保留：

- `tm memory search`
- `tm memory read`
- `tm task get`
- `tm task patch`

### 第三步

选定一个小范围文档集合，构建记忆系统 MVP。

### 第四步

让 Task Manager 跑通第一条“检索—读取—摘要—写状态”闭环。

### 第五步

增加一个高价值执行域，优先建议 `doc.*` 或 `sheet.*` 二选一。

### 第六步

补日志、回归测试与工具发现机制。

### 第七步

再把稳定工具接入 Codex 的正式工具层。

---

## 8. 一句话结论

**实施上应该“小步闭环”，架构上应该“协议先行”。**

也就是说：

- 一次只做一个最小任务闭环
- 但从一开始就把接口、命名、状态和扩展位设计好

这才是“不要一次搭得太大，但也保留可扩展性”的正确路径。
