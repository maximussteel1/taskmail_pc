# Task Manager 平台设计总纲（v0.2）

> 文档层级：Layer 3（未来平台上位设计）
>
> 重要说明：该文档描述未来 Task Manager 平台的目标边界与架构基线，不等同于 `mail_based_task_manager` 当前仓库的已实现范围。

更新日期：2026-03-14
状态：可作为平台级基线文档

## 1. 文档目的

本文用于定义 Task Manager 平台的顶层目标、边界、设计原则和总体架构。它不负责展开实现细节，而是作为后续接口规范、模块设计和开发计划的上位约束。

本文适合长期保留，修改频率应低于具体实现文档。

## 2. 背景与问题定义

当前工作中的大量时间消耗在数据处理、文档撰写、Excel 操作、资料检索、历史信息复用和邮件相关事务上。这些工作通常具备三个特点：

第一，任务流程相对稳定，但步骤繁琐，重复劳动占比高。

第二，核心信息大量沉淀在历史文档、表格、邮件和阶段性输出中，缺乏统一检索入口。

第三，大模型虽然擅长理解和生成，但并不适合直接承担长期记忆、批处理执行、版本管理、权限控制和审计追踪。

因此，需要建设一个以 Task Manager 为统一入口、以外部记忆系统为知识底座、以标准化工具为执行单元的工作平台。

## 3. 平台定位

本平台不是通用聊天助手，也不是单纯的文档库或脚本集合。

本平台的定位是：

**一个以 Task Manager 为调度中枢、以外部记忆系统为知识底座、以标准化工具为执行单元的个人工作自动化平台。**

平台的核心价值不在于“会聊天”，而在于围绕真实工作任务完成检索、规划、执行、生成、记录和复用。

## 4. 顶层目标

### 4.1 总体目标

构建一个可持续扩展、可追溯、可调用、可逐步自动化的工作平台，使 Task Manager 能够围绕历史文档、数据文件和工具系统，完成下列工作：

- 检索已有知识
- 读取并定位原始依据
- 调用工具完成处理
- 生成结构化输出
- 记录任务状态与执行轨迹
- 逐步替代高频重复性知识工作

### 4.2 阶段性目标

第一阶段，建立平台底座。

目标是明确平台边界、统一工具接入方式、建立外部记忆系统最小能力、建立任务状态管理，并跑通一个可运行的最小闭环。

第二阶段，建立核心生产能力。

目标是逐步接入文档生成、Excel 自动处理、数据分析、邮件辅助等高频能力，并使 Task Manager 能够在多工具之间完成可靠调度。

第三阶段，建立规模化复用能力。

目标是引入工作流模板、成熟任务半自动执行、任务评估与回归验证，使平台从“可用”走向“长期稳定可用”。

## 5. 非目标

为了避免平台方向失控，当前版本明确以下事项不属于首期目标：

- 不追求建设一个通用、无边界的 AI 系统。
- 不追求一次性清洗全部历史文档。
- 不依赖单条提示词解决长期记忆问题。
- 不让模型直接承担大规模文件读写、计算和状态管理。
- 不在首期追求全自动闭环执行。

平台首要追求的是结构正确、边界清晰、接口稳定，而不是功能数量。

## 6. 核心设计原则

### 6.1 外部化原则

长期记忆、业务事实、任务状态、日志和原始文件必须外部化存储。模型不承担这些信息的最终保存职责。

### 6.2 工具化原则

所有核心能力必须以标准化工具暴露，而不能依赖临时提示词或人工默契。

### 6.3 协议优先原则

优先冻结输入输出协议和职责边界，再讨论具体实现。只要协议稳定，后续可以独立演进底层实现。

### 6.4 可追溯原则

平台输出的结论、摘要、建议和文档，应尽量回溯到原始文档、原始片段、任务状态和执行记录。

### 6.5 渐进式建设原则

优先完成最小可用闭环，再扩展场景和能力。先保证“稳定做对”，再追求“覆盖更多任务”。

### 6.6 模型与程序分工原则

模型负责理解、规划、归纳、生成和选择。

程序工具负责检索、计算、转换、读写、存储、执行和审计。

## 7. 总体架构

平台总体分为五层。

### 7.1 交互与调度层

核心组件为 Task Manager。

职责包括接收请求、理解任务目标、规划执行步骤、选择工具、协调多步调用、汇总结果并更新任务状态。

这一层是平台的统一入口，但不是长期记忆的承载位置。

### 7.2 规则与策略层

用于存放长期稳定的行为规则和流程约束，包括平台角色、工具调用原则、输出要求、权限边界、固定工作流说明等。

这一层解决的是“系统应当如何做”，而不是“系统知道什么”。

### 7.3 外部记忆层

用于承载历史文档、解析结果、索引、元数据和引用定位信息。

其作用不是做普通文件夹管理，而是建立一个可检索、可读取、可引用的工作知识库。

### 7.4 工具执行层

用于承载所有具体执行能力。后续所有能力都应作为标准化工具接入这一层，例如 memory、task、doc、sheet、data、email 等域。

### 7.5 存储与记录层

用于保存原始文件、索引数据、任务状态、日志、缓存、版本信息和执行记录。

没有这一层，平台只能是一次性对话系统；有了这一层，平台才具备持续工作能力。

## 8. 核心能力域

从平台视角看，后续所有能力最终都应落在四个基础域上：

### 8.1 记忆能力

解决“系统知道什么”的问题，包括文档入库、解析、索引、检索、读取和定位。

### 8.2 状态能力

解决“系统做到哪里”的问题，包括任务状态、上下文摘要、中间结果和执行轨迹的保存。

### 8.3 执行能力

解决“系统如何完成任务”的问题，包括文档处理、Excel 处理、数据分析、格式转换、邮件处理等。

### 8.4 调度能力

解决“先做什么、后做什么、调用什么”的问题。这是 Task Manager 的核心职责。

## 9. 工具接入策略

考虑到后续会持续增加大量工具，平台必须尽早统一结构。

平台应遵循以下策略：

第一，所有工具都应有清晰边界，尽量只承担单一职责。

第二，所有工具都应具有稳定的输入输出 schema。

第三，所有工具都应能被 Task Manager 程序化调用，而不是只适合人手执行。

第四，工具日志与工具结果必须分离。

第五，工具失败时必须返回可解释错误，而不是模糊文本。

在接入方式上，CLI 可以作为底层执行适配层，但平台真正依赖的是标准化工具契约，而不是命令行形式本身。

换句话说：

**CLI 是执行层，JSON schema 才是正式接口。**

## 10. 外部记忆系统的定位

外部记忆系统是平台首期最重要的基础能力。

它的首期目标不是建设一个完美知识库，也不是一次性吃入全部历史文档，而是提供一个足够稳定、可被 Task Manager 调用的最小记忆接口层。

其最小能力至少应包括：

- 按条件和关键词检索候选文档片段
- 读取指定文档或片段内容
- 返回文档标识和片段标识
- 提供原文定位信息以支持引用

其中，类型、时期、关键词可以作为首版过滤条件，但它们只是入口，不是记忆系统的全部。

如果没有读取和定位能力，系统只能“找到可能相关的文档”，无法支撑后续的摘要、复用和引用。

## 11. Task Manager 的定位与边界

Task Manager 不是知识库，不是工具堆，也不是文档存储系统。

Task Manager 的定位是：

**平台级任务调度器。**

它的职责包括：

- 接收自然语言请求
- 识别任务类型
- 判断需要调用的能力
- 发起检索与读取
- 组织执行顺序
- 汇总结果
- 更新状态
- 输出最终结果

它不负责长期存储业务事实，也不应把可变知识压进提示词中。

## 12. 模型、记忆与工具的关系

平台采用如下分工：

- 规则由规则层提供。
- 事实由外部记忆层提供。
- 执行由工具层完成。
- 状态由状态系统保存。
- 调度由 Task Manager 完成。
- 模型负责在给定规则和给定工具范围内做出规划与生成。

因此，平台稳定性的核心不在于提示词是否足够长，而在于规则、工具、记忆和状态是否已经外部化并形成稳定契约。

## 13. 演进路线

### 13.1 第一阶段：基础设施期

目标是完成平台底层结构的固定，包括：

- 明确顶层目标和非目标
- 统一工具命名与接口规范
- 确立外部记忆系统最小接口
- 确立任务状态接口
- 跑通一个最小闭环场景

### 13.2 第二阶段：核心生产期

目标是让平台真正承担高频工作，包括：

- 扩展文档库和索引范围
- 增加文档摘要与汇总工具
- 增加 Excel 和数据处理工具
- 增加邮件工具和模板化流程

### 13.3 第三阶段：规模化复用期

目标是让平台从“可用”走向“长期可用”，包括：

- 多项目规则管理
- 工作流模板
- 任务评估和回归测试
- 权限、审计和版本机制完善
- 高成熟度任务半自动或全自动执行

## 14. 关键风险与控制思路

### 14.1 过度依赖提示词

风险：业务知识和工作规则大量堆进提示词，导致系统脆弱且难以维护。

控制：规则、记忆、状态和工具全部外部化。

### 14.2 工具数量增长后失控

风险：工具越来越多，但命名、职责和返回结构不统一，导致调度复杂度失控。

控制：先冻结命名体系、schema 和错误结构，再扩展工具数量。

### 14.3 知识库建设过重

风险：一开始试图整理全部历史文档，导致项目在清洗阶段长期停滞。

控制：先选高价值文档集合做试点，跑通闭环后逐步扩库。

### 14.4 调度层先于底层能力过度细化

风险：在接口未定之前过度设计 Task Manager，后续反复返工。

控制：先定义基础能力契约，再细化调度逻辑。

## 15. 一句话定义

**本平台是一个以 Task Manager 为调度中枢、以外部记忆系统为知识底座、以标准化工具为执行单元的个人工作自动化平台。**
