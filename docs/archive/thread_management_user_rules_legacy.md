## 1. 目的

本文从用户视角回答一个问题：

> 当用户通过邮件持续和同一个工作目录交互时，应该如何开启、继续、结束、恢复、查看和处理线程。

本文不讨论底层实现细节，不要求用户理解 native backend session、CLI resume 或本地状态文件结构。

用户只需要理解：

- 什么算“同一件事”
- 什么时候应该直接回复
- 什么时候应该新开线程
- 什么时候应该结束线程
- 什么时候应该恢复线程
- 什么时候应该手动停止或排查卡死线程

## 2. 用户心智模型

面向用户的推荐心智模型如下：

- `线程`：一条长期存在的任务线，代表“同一件事”
- `运行`：线程中的一次执行尝试
- `工作目录`：线程所关联的 `repo + workdir`
- `结束`：把线程从“当前活跃工作集”中拿掉，但不删除历史
- `恢复`：继续已经存在的线程，而不是强制新开一条线程

关键原则：

- 用户不需要关心底层 native session 是否还活着
- “继续线程”是用户语义；系统可以在后台决定是 native resume 还是 fresh continuation
- `DONE`、`FAILED`、`KILLED` 只是**上一轮运行结果**
- 线程本身是否还算“正在处理”，应由用户控制，而不是由单次运行结果自动决定
- `[DONE]` 同时也是用户已经收到结果的明确回执；这封回执邮件应保留，不应被后续状态邮件删除

## 3. 基本规则

### 3.1 同一件事，继续同一线程

如果用户仍在处理同一件事，应继续回复原线程。

这包括：

- 补充需求
- 上传新截图或文件
- 回答系统问题
- 要求继续处理
- 对上一次结果做修正

### 3.2 不同的事，开新线程

如果用户要处理的是另一件事，即使工作目录相同，也应新开线程。

典型例子：

- 同一个仓库里修两个不相关问题
- 同一个 App 里同时做“登录页改版”和“支付 Bug 修复”
- 同一个目录里分别维护两个独立实验分支

### 3.3 结束不是删除

用户主动“结束线程”，表示：

- 这条线程暂时不再算活跃工作
- 默认不再出现在活跃线程列表里
- 历史记录、产物、摘要和邮件链仍然保留

结束不表示：

- 删除 thread archive
- 删除工作目录中的改动
- 永久禁止再次继续

### 3.4 恢复不是重开

用户“恢复线程”，表示继续原来的那件事。

系统应优先保持：

- 相同线程身份
- 相同工作目录
- 相同历史上下文

系统不应要求用户先判断：

- native backend session 还在不在
- 是否必须重新建会话
- 是否必须手动复制旧任务描述

### 3.5 `DONE` 是回执，不是可随时覆盖的瞬时进度

当线程发出 `[DONE]` 时，用户已经拿到了这一轮的明确回复。

因此推荐规则是：

- `[DONE]` 邮件应保留
- 后续如果线程继续，应发送新的状态邮件，但不删除之前的 `[DONE]`
- `[DONE]` 表示“这一轮完成了”，不表示“这条线程被自动结束了”

## 4. 推荐命令与动作语义

本文推荐以下用户动作。

其中一部分当前已经存在，另一部分是建议新增的线程管理控制面。

### 4.1 已有动作

- 首封新任务：`[OC]` / `[CX]`
- 当前线程继续：直接 reply
- 暂停当前线程：`/pause`
- 恢复暂停线程：`/resume`
- 查看当前线程状态：`/status`
- 重跑当前线程：`/rerun`
- 停止当前运行进程：`/kill`
- 在当前 reply 链里新开 session：`/new`
- 查看当前 workspace 的 session 列表：`/sessions`

### 4.2 建议新增动作

- 结束当前线程：`/end`
- 全局线程管理入口：首封管理邮件 `Subject: [SESSIONS]`
- 全局线程管理操作：
  - `list active`
  - `list all`
  - `list stuck`
  - `status <thread_id>`
  - `resume <thread_id>`
  - `end <thread_id>`
  - `kill <thread_id>`

### 4.3 用户不应理解为不同概念的动作

以下语义应在用户层尽量统一：

- “继续”与“恢复”都属于继续原线程
- “恢复”主要用于已经暂停或已结束的线程
- “继续当前线程”不应要求用户理解 native resume

## 5. 线程生命周期规则

面向用户，线程应有两条彼此独立的轴：

### 5.1 运行状态轴

运行状态描述上一轮执行的结果：

- `queued`
- `running`
- `waiting_user`
- `paused`
- `done`
- `failed`
- `killed`

### 5.2 生命周期轴

生命周期描述这条线程是否仍属于当前活跃工作集：

- `active`
- `ended`

推荐规则：

- 新线程默认是 `active`
- `/end` 把线程改成 `ended`
- `/resume` 可以把 `ended` 线程拉回 `active`
- `done` 不自动改成 `ended`
- `failed` 不自动改成 `ended`
- `killed` 不自动改成 `ended`
- `paused` 不自动改成 `ended`

理由：

- 用户经常会在一次 `DONE` 之后继续微调
- `FAILED` 之后继续修复是常见路径
- `KILLED` 只代表停止了当前进程，不代表用户放弃这件事
- 邮件场景存在天然长间隔，不应使用“沉默即结束”的假设

## 6. 场景化用户操作规则

本节按用户真实场景给出推荐操作。

### 6.1 场景 A：开始一件新任务

用户动作：

- 发首封新邮件
- `Subject: [OC] ...` 或 `Subject: [CX] ...`

系统预期：

- 创建新线程
- 创建新 session
- 进入活跃工作集

### 6.2 场景 B：继续同一件事

用户动作：

- 直接回复该线程最近一封状态邮件

系统预期：

- 继续同一线程
- 优先复用已有上下文
- 必要时在后台自动改走 fresh continuation

用户不需要：

- 重新粘贴完整任务描述
- 手动判断是否该用 `/new`

### 6.3 场景 C：同一工作目录里开启另一件事

用户动作：

- 发新的首封任务邮件
- 或在现有 reply 链里显式使用 `/new`

系统预期：

- 创建另一条新线程
- 新线程和旧线程共享同一 `repo/workdir`，但互不混淆

规则：

- 不能因为工作目录相同，就把不同的事情自动并入同一线程
- 不应依赖“标题相似”自动命中旧线程

### 6.4 场景 D：我这条线程先不做了，但未来可能继续

用户动作：

- 在该线程里发送 `/end`

系统预期：

- 线程退出活跃工作集
- 历史和产物仍保留
- 以后仍可恢复

这个动作适用于：

- 当前阶段完成
- 等待外部决定
- 用户要把注意力转到别的线程
- 用户只是想把工作集收干净

### 6.5 场景 E：我想继续一条之前结束的线程

用户动作：

- 回复该线程最近一封邮件并发送 `/resume`
- 或在全局管理入口发送 `resume <thread_id>`

系统预期：

- 该线程重新进入活跃工作集
- 系统继续原任务线

推荐规则：

- `/resume` 的用户语义是“继续这条线程”
- 不要求底层一定具备原生 native session

### 6.6 场景 F：线程当前在等待我回答问题

用户动作：

- 直接回复答案
- 如果线程此前已被 `/pause`，则使用 `/resume` 并附上答案

系统预期：

- 单题直接按答案处理
- 多题按结构化答案处理
- 不应该强迫用户在普通 waiting state 下额外理解线程切换问题

### 6.7 场景 G：线程被我主动暂停了

用户动作：

- 想继续时发送 `/resume`

系统预期：

- 线程恢复为可继续状态
- 普通 reply 不应隐式解除 `paused`

原因：

- `paused` 是显式冻结态
- 它的目标就是避免误触恢复

### 6.8 场景 H：线程显示 DONE，但我还想继续改

用户动作：

- 直接 reply，补充新要求

系统预期：

- 继续同一线程
- 生成新的 run

规则：

- `DONE` 不代表线程自动结束
- 它只代表上一轮运行成功完成
- `[DONE]` 邮件本身应继续保留，作为用户已收到结果的回执

### 6.9 场景 I：线程显示 FAILED，但我还想继续修

用户动作：

- 直接 reply，说明下一步
- 或使用 `/resume`

系统预期：

- 继续同一线程
- 如果 native resume 不可用，系统自动改走本地 recap 驱动的 fresh continuation

规则：

- 用户层不需要理解“这次是 native resume 还是 fresh continuation”

### 6.10 场景 J：线程显示 KILLED

用户动作：

- 如果只是想停掉当前进程并稍后再看，可先 `/end`
- 如果要继续同一件事，可发送 `/resume`
- 如果要从较清晰的起点重开本轮执行，可发送 `/rerun`

系统预期：

- `kill` 只终止当前运行，不自动删除线程
- 后续操作仍围绕同一线程进行

### 6.11 场景 K：我忘了自己开过哪些线程

用户动作：

- 发送首封管理邮件 `Subject: [SESSIONS]`
- 正文使用：
  - `list active`
  - `list all`
  - `list stuck`

系统预期：

- 返回线程列表，而不是只返回当前 workspace
- 默认优先展示 `active`
- `list all` 同时显示 `active` 与 `ended`

列表中建议至少展示：

- `thread_id`
- 标题或 session name
- `repo/workdir`
- lifecycle
- 当前运行状态
- 最后更新时间
- 最近摘要

### 6.12 场景 L：我知道线程 ID，想远程管理它

用户动作：

- 向 `Subject: [SESSIONS]` 的管理线程发送：
  - `status <thread_id>`
  - `resume <thread_id>`
  - `end <thread_id>`
  - `kill <thread_id>`

系统预期：

- 执行动作后，结果仍回写到目标线程自己的邮件链
- 管理线程只承担控制面，不应吞掉业务线程的后续状态流

### 6.13 场景 M：我感觉线程卡死了

用户动作：

1. 先查看管理列表里的 `list stuck`
2. 或查看 `status <thread_id>`
3. 如果确认需要人工干预，再执行 `kill <thread_id>`
4. 之后根据需要：
   - `/resume`
   - `/rerun`
   - `/end`

系统预期：

- 系统应该能把“正常运行”和“疑似卡死”区分开
- 用户应该能看见线程最近一次有进展的时间

## 7. 卡死与异常判断规则

用户需要知道“线程是不是不工作了”，因此系统应给出明确的健康判断。

推荐的用户可见健康标签：

- `normal`
- `stale`
- `suspected_stuck`
- `orphaned`

建议语义：

- `normal`：线程状态正常
- `stale`：长时间未更新，但当前不一定有运行
- `suspected_stuck`：线程仍显示 `running`，但长时间没有进展
- `orphaned`：线程记录为 `running`，但宿主进程或工作进程已不存在

用户动作建议：

- `stale`：先看是否其实已经不需要处理，可直接 `/end`
- `suspected_stuck`：优先人工检查，再决定是否 `/kill`
- `orphaned`：应允许用户直接 `/kill` 或 `/resume`

## 8. 推荐默认列表规则

为了降低“忘记关线程”的成本，默认列表应遵守以下规则：

- 默认只展示 `active`
- 默认按 `updated_at` 倒序
- 默认突出 `running`、`waiting_user`、`suspected_stuck`
- `ended` 线程默认折叠，不占用主视图

这样用户可以：

- 把“当前在做的事”看成一个活跃工作集
- 把“历史线程”留在 `all` 视图里，需要时再恢复

## 9. 不推荐的用户语义

以下语义不推荐进入最终协议：

### 9.1 不推荐“DONE 自动结束线程”

原因：

- 大量实际任务会在 `DONE` 后继续微调
- 这会让用户每次追加小修都不得不重新开线程

### 9.2 不推荐按沉默时间自动结束线程

原因：

- 邮件往返天然有长间隔
- 用户不应该因为几天没回邮件，就被系统擅自改变线程语义

### 9.3 不推荐按标题自动把新邮件并入旧线程

原因：

- 同工作目录、多相似标题是常见情况
- 自动归并极易把不同任务线混在一起

### 9.4 不推荐把“恢复线程”定义成“必须复用 native backend session”

原因：

- 用户关心的是继续同一件事
- native session 是否可用应由系统内部处理

## 10. 当前实现与目标规则之间的差距

截至 2026-03-17，当前代码已经具备的能力：

- 同 workspace 支持多个 session
- reply 可以继续当前线程
- `/new` 可以在 reply 链里显式新开 session
- `/pause`、`/resume`、`/status`、`/rerun`、`/kill` 已存在
- `/sessions` 已能列出当前 workspace 的 session

当前仍缺失的能力：

- `/end`
- 生命周期轴 `active/ended`
- 全局管理入口 `Subject: [SESSIONS]`
- 面向用户的全局线程列表与显式目标控制
- `stuck/orphaned` 用户可见判断
- “继续线程但不强依赖 native resume”的正式用户协议收口

因此，建议后续协议与实现优先级为：

1. 增加 `/end`
2. 增加 `active/ended` 生命周期轴
3. 增加全局 `SESSIONS` 管理入口
4. 增加 stuck/orphaned 可观测性
5. 收口 continuation / resume 的上下文重建策略

## 11. 面向用户的最简操作清单

如果只给用户一页最简规则，应压缩成下面几条：

- 同一件事：直接回复原线程
- 不同的事：新开线程
- `[DONE]` 回执会保留；后续继续同一线程时，不会因为新状态而抹掉它
- 暂时不做了：`/end`
- 要继续以前那条：`/resume`
- 只是暂停误触保护：`/pause`
- 想停掉当前进程：`/kill`
- 想从头重跑当前任务线：`/rerun`
- 忘了自己开了哪些线程：发 `Subject: [SESSIONS]`，正文写 `list active`
- 怀疑线程卡死：先 `list stuck`，再决定是否 `kill <thread_id>`

## 12. 结论

推荐的线程管理用户语义是：

- 线程长期存在
- 运行结果不等于线程终结
- `[DONE]` 是明确结果回执，应保留
- 用户可以主动结束线程
- 结束后仍可恢复
- 同一工作目录允许多个线程并存
- 用户应有全局入口查看 active / ended / stuck 线程
- 用户不需要理解 native session，只需要理解“是不是继续同一件事”
